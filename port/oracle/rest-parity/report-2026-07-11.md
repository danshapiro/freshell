## Comparison contract

- **Requests:** byte-identical against both servers (same token, same paths,
  same bodies). Where both servers share a filesystem side-effect location
  (screenshot outputs under the shared checkout cwd), per-side hooks reset the
  filesystem between the two sends — the requests themselves never differ.
- **Compared:** HTTP status; headers-of-interest (`content-type`,
  `cache-control`, `x-frame-options`, `content-security-policy`,
  `content-security-policy-report-only`, `x-qa-marker`); body as normalized
  deep-equal (JSON), scrubbed text, or sha256 of raw bytes. Body
  representation is content-type-independent so a header divergence cannot
  skew the body comparison.
- **WS cases** compare the full inbound frame transcript + close code/reason.

## Environment/harness notes (honest disclosures)

1. **cwd = checkout for both servers.** The §5.1/§5.2 recipes run from the
   checkout; this is load-bearing for the ORIGINAL — its extension registry
   scans `$CWD/extensions` (`server/index.ts:224-228`) and the
   `availableClis` detection specs derive from those manifests. A first sweep
   run with an isolated cwd made the original return an EMPTY extension
   registry and `availableClis: {}` — a harness artifact, corrected.
2. **Checkout `.env` neutralized** for the node server via
   `DOTENV_CONFIG_PATH` pointing at a nonexistent file (explicit env always
   wins anyway; this prevents asymmetric keys like `GEMINI_API_KEY` flipping
   `featureFlags.aiEnabled` on one side only). `KILROY_ENABLED=0` pinned on
   both sides.
3. **Session seeding:** both homes were seeded identically from
   `test/fixtures/sessions/healthy.jsonl` (per task charter) **plus one
   synthesized realistic Claude session**. Reason: the repair fixtures are not
   valid coding-cli sessions (plain-string `message` fields, non-UUID
   `session_id`) — verified empirically with an at-boot probe: the node
   original never surfaces them in `/api/session-directory` under any flag.
   Without a realistic session the seeded entries/filters/revision path would
   compare empty-vs-empty and prove nothing. The fixture seeding is retained
   and pinned a real divergence (R10b: the rust port DOES surface the fixture
   under `includeNonInteractive`/`includeEmpty`).
4. **`settings.updated-broadcast.put` row caveat:** both servers emit a
   `settings.updated`-typed frame during the post-hello handshake snapshot;
   the PUT-row broadcast capture picked up that handshake frame (values are
   pre-PUT on both sides). The clean PUT-triggered-broadcast comparison is the
   **`.patch` row** (observers cleared between). The row verdicts stand as
   recorded; interpret the `.put` row as handshake-snapshot parity evidence.

## Systematic divergence families

### S1 — JSON `Content-Type` charset suffix (63 cases where it is the ONLY diff)

- node/Express: `application/json; charset=utf-8`
- rust: `application/json`

Same JSON bodies after normalization. One root cause, repeated across nearly
every JSON endpoint. Every affected row is still recorded as DIVERGENCE (the
comparison contract includes `content-type`); adjudication of whether the port
must emit the charset suffix is out of scope for this task.

### S2 — Static/SPA `Content-Type` differences

| Case | node | rust |
|---|---|---|
| `spa.root`, `spa.deeplink`, `spa.nonasset.missing` (SPA html) | `text/html; charset=UTF-8` | `text/html; charset=utf-8` (case) |
| `spa.asset.real` (`.js` asset — bytes sha256-equal: `184c0414…`) | `application/javascript; charset=UTF-8` | `text/javascript; charset=utf-8` |
| `spa.asset.missing` (404 body `Not found`) | `text/html; charset=utf-8` | `text/plain; charset=utf-8` |

Bodies were byte-identical in all SPA cases (index.html, assets, 404 text);
only the header differs. Cache-control headers (no-store for index, immutable
for hashed assets) MATCHED.

### S3 — WS close semantics (3 cases: `ws.badtoken`, `ws.notoken`, `ws.protomismatch`)

Error FRAMES are identical (`NOT_AUTHENTICATED`/`Invalid token`,
`PROTOCOL_MISMATCH`/`Expected protocol version 7. Please reload the page.`).
But the node original then closes with a proper WebSocket close frame
(code **4001** reason `Invalid token`; code **4010** reason
`Protocol version mismatch`), while the rust port drops the connection without
a close frame — the client observes **1006** (abnormal closure), empty reason.

## Substantive findings (R1–R14)

### R1 — `PUT /api/settings` is unimplemented on the rust port (405, empty body)

The original treats PUT and PATCH identically (`server/settings-router.ts:151-152`).
The port returns **405 with an empty body** for every PUT — including all
validation-error requests, so the enum/type/strict-schema error shapes could
only be exercised via node:

| Case | node | rust |
|---|---|---|
| `settings.put.happy` (`{safety:{autoKillIdleMinutes:20}}`) | 200 + updated settings | **405**, empty |
| `settings.put.enum-invalid` (`editor.externalEditor='bogus'`) | 400 `{error:'Invalid request', details:[zod invalid_value…]}` | **405**, empty |
| `settings.put.type-invalid`, `.nested-enum-invalid`, `.unknown-key`, `.client-key-rejected` | 400 + zod details | **405**, empty |
| `settings.put.agentchat-migrated` | 400 `agentChat settings have been migrated; use freshAgent` | **405**, empty |

### R2 — rust settings PATCH does not reach the served/persisted settings state

Observed chain (all from the results JSON, in execution order):

1. `settings.patch.happy` (`autoKillIdleMinutes:25`): **both 200**, rust
   response body reflects 25 and rust broadcasts `settings.updated` with 25
   (`settings.updated-broadcast.patch`).
2. `settings.get.after-put`: node `safety.autoKillIdleMinutes=25`, **rust=15
   (default)** — the GET does not see the patch the PATCH response reported.
3. `settings.configjson-shape` (direct scratch-home read): node config.json
   contains the full settings tree + `projectColors`/`recentDirectories`/
   `sessionOverrides`/`terminalOverrides`/`serverSecrets`; **rust config.json
   still contains only the pre-seeded `network` block** — no patched settings
   persisted.
4. `settings.persist.restart` (after full restart): node still `25` +
   `allowedFilePaths:[]`; **rust back to `15`, no `allowedFilePaths`** — the
   patch did not survive restart.

### R3 — `allowedFilePaths` sandbox is not enforced by the rust port (security-relevant)

After `settings.patch.sandbox-on` (`allowedFilePaths:['~/qa-files']`, both 200,
both responses echoing the sandbox):

| Case | node | rust |
|---|---|---|
| `files.sandbox.allowed` (inside) | 200 | 200 (parity) |
| `files.sandbox.denied-read` (`~/outside.txt`) | **403** `{error:'Path not allowed'}` | **200 + file content** |
| `files.sandbox.denied-complete` (`~/`) | **403** | **200 + directory listing** |
| `files.sandbox.denied-write` (`~/outside2.txt`) | **403** | **200, file written** |
| `files.sandbox.denied-mkdir` (`~/qa-outside-dir`) | **403** | **200, dir created** |

Given R2, it is not distinguishable from the REST surface alone whether
enforcement is missing or the patched sandbox config simply never reaches the
enforcement path — either way the served behavior diverges and reads/writes
outside the sandbox succeed on the port.

### R4 — `codingCli.knownProviders` not tracked by the rust port

Node populates `knownProviders: ["claude","codex","gemini","kimi","opencode"]`
(from the extension manifests) in GET/PATCH responses, broadcasts, and
persisted config; rust always reports `knownProviders: []`. Affects
`settings.get.default`, `.get.after-put`, `.patch.*`, both broadcast rows,
`settings.persist.restart`, `settings.configjson-shape`.

### R5 — `GET /api/version` `updateCheck`: rust is always `null`

Node returns a live update-check object (masked `<OPAQUE>` — live GitHub
data); rust returns `null`. `currentVersion` itself is normalized (`version`
family). Cases: `version.happy`, `version.cookie-auth`.

### R6 — `GET /api/extensions/:name` unimplemented on the rust port

- `extensions.single` (`/api/extensions/claude`): node 200 + full
  `ClientExtensionEntry`; rust **404 `{error:'Not found'}`**.
- `extensions.single.missing` (`/api/extensions/does-not-exist`): both 404,
  but node `{error:"Extension not found: 'does-not-exist'"}` vs rust
  `{error:'Not found'}`.
- The LIST route (`extensions.list`, the §7.C "5-entry registry") matched
  exactly (S1-only) once the recipe cwd was used.

### R7 — `POST /api/files/mkdir` `existed` flag semantics

rust `{created:true, existed:true}`. First creation and the 409-over-file

### R11 — `POST /api/screenshots` error precedence

The rust port checks for a screenshot-capable client BEFORE name/path
validation and the no-overwrite conflict; the original validates first:

| Case | node | rust |
|---|---|---|
| `shots.emptyname` | **400** `name required` | **503** `No screenshot-capable UI client connected` |
| `shots.sep-in-name` | **400** `name must not contain path separators` | **503** (same) |
| `shots.dup409` (pre-existing output, no overwrite) | **409** `output file already exists (use --overwrite)` | **503** (same) |

With a participating client connected, the full round-trip MATCHED: ok
envelope (incl. `path`/`width`/`height`/focus flags), the `ui.command`
frame, written PNG bytes (sha256-equal), overwrite path, and the 422
failure envelope (`qa-forced-failure`). Scope-validation 400s
(`shots.badscope`, `.pane-no-paneid`, `.tab-no-tabid`) matched (S1-only),
as did the no-client 503 for a valid request (`shots.noclient503`).

### R12 — auth vs routing precedence for unknown `/api` routes

`{error:'Unauthorized'}`** (auth middleware runs first); rust **404
`{error:'Not found'}`** — the port discloses route existence to
unauthenticated callers. With credentials both return the same JSON 404
(`api.unknown-route` PASS).
