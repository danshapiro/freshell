## §7.H.4 — Tauri window-state restore across restart (rust `create_main_window`, lib.rs WIP)

Evidence (all with the rebuilt release binary at this commit; graceful smoke-timer exits ONLY —
never `xdotool windowclose`; logs + desktop.json quoted; `tauri-ws-r16-run*.log`,
`ws-r16-run*.png` committed here):

- **Run 1 (save path)**: fresh `FRESHELL_HOME` → boot → `xdotool windowmove 350 220` +
  `windowsize 900 650` → after the 500ms debounce, `desktop.json.windowState` =
  `{x:356, y:247, width:900, height:650, maximized:false}` — exactly the window's resulting
  inner-logical bounds. Graceful exit preserved the file.
- **Run 2 (restore path)**: relaunch → window restored at **exactly 900x650**; requested position
  (356,247); observed frame at (362,274) — see the ENV note below. Re-save recorded the actual
  bounds (honest capture). `ws-r16-run2-restored.png`.
- **Run 4 (off-screen clamp)**: `desktop.json` hand-written to `x:-5000, y:-4000, 800x600` →
  relaunch → boot log: `persisted window bounds clamped on-screen (-5000,-4000 800x600 ->
  320,180 800x600)`; window on-screen at 800x600 (`ws-r16-run4-clamped.png`). The clamp is the
  one deliberate, ledgered extra vs the reference (Electron restores off-screen verbatim —
  documented in `window_state.rs`).

**ENV-LIMITED (WSLg Weston position offset, with control):** on this host's compositor EVERY window
move request lands +(6,27) from the requested position — including a plain
`xdotool windowmove 350 220`, which landed the frame at (356,247), the same +(6,27) shift the
restore path exhibits (356,247 requested → 362,274 observed; run 3 repeat: 362,274 → 368,301).
Size restore is EXACT; the position drift is compositor-imposed on any client (the Electron
reference issues the same X11 configure requests through the same Weston WM), not a port defect.
On a standard WM that honors placement, save→restore is a fixed point; re-verify there when
available. The xdotool control is the proof this is display-server behavior, not `lib.rs` math.
