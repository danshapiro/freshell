# Issue #199 -- Eliminate Redundant .jsonl File Reads via Path Resolution and Parsed Content Caching

## Problem

`loadSessionHistory()` in `server/session-history-loader.ts` performs a brute-force directory scan
across `~/.claude/projects/*/` and `~/.claude/projects/*/*/` every time a session's chat history
is requested. On a machine with ~216 project directories this is extremely wasteful because:

1. The `CodingCliSessionIndexer` already maintains a `sessionKeyToFilePath` map (populated by
   chokidar watchers) that can resolve `sessionId -> filePath` in O(1).
2. Every call to `loadSessionHistory()` re-reads and re-parses the entire `.jsonl` file even
   when the file has not changed since the last read.
3. The agent-timeline endpoint makes a separate HTTP request per turn body, each re-reading
   the full file.

Three call sites in `ws-handler.ts` (lines ~1714, ~1887, ~1937) plus the agent-timeline service
all invoke `loadSessionHistory()` independently, often for the same session within seconds.

## Solution: Three Layers

### Layer 1 -- Path Resolution Cache

**Goal:** Replace the O(N) directory scan with an O(1) index lookup.

**Mechanism:** Inject `codingCliIndexer.getFilePathForSession()` into `loadSessionHistory()`
as an optional `resolveFilePath` dependency. When provided, `loadSessionHistory()` calls the
resolver first. If the resolver returns a path, skip the scan entirely. If it returns
`undefined`, fall back to the existing scan (needed at startup before the indexer is ready,
and for non-Claude providers).

**Changes:**

| File | Change |
|------|--------|
| `server/session-history-loader.ts` | Add optional `deps.resolveFilePath` parameter to `loadSessionHistory()`. When provided and returning a path, `readFile` that path directly. Keep the scan as fallback. |
| `server/index.ts` | When wiring `createAgentTimelineService`, pass a `loadSessionHistory` that uses the indexer's resolver: `(sessionId) => loadSessionHistory(sessionId, undefined, { resolveFilePath: (id) => codingCliIndexer.getFilePathForSession(id, 'claude') })` |
| `server/ws-handler.ts` | Accept a `deps.resolveSessionFilePath` in the constructor (or import from the same wired helper). Thread it into `loadSessionHistory()` calls at lines ~1714, ~1887, ~1937. |

**Signature change for `loadSessionHistory`:**

```ts
export interface LoadSessionHistoryDeps {
  resolveFilePath?: (sessionId: string) => string | undefined
}

export async function loadSessionHistory(
  sessionId: string,
  claudeHome?: string,
  deps?: LoadSessionHistoryDeps,
): Promise<ChatMessage[] | null>
```

When `deps.resolveFilePath` returns a string, the function:
1. Validates the session ID (path traversal check, same as today).
2. Reads the file at the resolved path directly.
3. Returns `extractChatMessagesFromJsonl(content)`.
4. On ENOENT/read error, falls through to the existing scan.

### Layer 2 -- Parsed Content Cache

**Goal:** Avoid re-reading and re-parsing `.jsonl` files that have not changed.

**New module: `server/session-content-cache.ts`**

An LRU cache keyed by `filePath` (absolute, normalized). Each entry stores:

```ts
interface SessionContentCacheEntry {
  mtimeMs: number
  size: number
  messages: ChatMessage[]
  byteSize: number       // Approximate memory footprint for size-aware eviction
  cachedAt: number        // For debugging/metrics
}
```

**Class: `SessionContentCache`**

The cache imports `extractChatMessagesFromJsonl` from `./session-history-loader.js`
directly. This keeps parsing co-located with caching and avoids requiring callers to
supply a parse function.

```ts
import { extractChatMessagesFromJsonl, type ChatMessage } from './session-history-loader.js'

export class SessionContentCache {
  private cache: Map<string, SessionContentCacheEntry>  // insertion-order = LRU
  private inflight: Map<string, Promise<ChatMessage[] | null>>  // request coalescing
  private totalBytes: number
  private readonly maxBytes: number  // ~100 MB default

  constructor(options?: { maxBytes?: number })

  /**
   * Get parsed messages for a file path.
   * - stat() to check mtime+size
   * - If cache hit and mtime+size match, return cached messages
   * - If cache miss or stale, read file, parse, store, return
   * - Concurrent calls for the same path coalesce into one read
   * - On read error (ENOENT, permission), return null and evict
   */
  async get(filePath: string): Promise<ChatMessage[] | null>

  /**
   * Invalidate a specific entry (called when chokidar detects a change).
   */
  invalidate(filePath: string): void

  /**
   * Clear the entire cache.
   */
  clear(): void

  /**
   * Stats for debugging/logging.
   */
  stats(): { entries: number; totalBytes: number; maxBytes: number }
}
```

**Eviction strategy:**

- LRU via `Map` iteration order: on insert, if `totalBytes > maxBytes`, delete
  entries from the front of the map until under budget.
- `byteSize` is approximated as `2 * jsonStringLength` (V8 uses UTF-16 internally,
  so string byte cost is ~2x the character count) plus a fixed overhead per entry.
- The 100 MB budget is configurable via constructor option and env var
  `FRESHELL_SESSION_CACHE_MAX_MB`.

**Request coalescing:**

```ts
async get(filePath: string): Promise<ChatMessage[] | null> {
  // Check inflight first
  const existing = this.inflight.get(filePath)
  if (existing) return existing

  const promise = this._doGet(filePath)
  this.inflight.set(filePath, promise)
  try {
    return await promise
  } finally {
    this.inflight.delete(filePath)
  }
}
```

**Adversarial handling:**

- File deleted between stat and read: catch ENOENT, return null, evict entry.
- File mutated between stat and read: the read returns slightly stale data for this
  one call; the next call will see a different mtime and re-read. This is acceptable
  because `.jsonl` files are append-only in practice.
- Rapid mtime churn (writer appending lines quickly): each call stats the file;
  if mtime changed, re-reads. The coalescing prevents multiple concurrent re-reads.

**Integration into `loadSessionHistory`:**

```ts
export async function loadSessionHistory(
  sessionId: string,
  claudeHome?: string,
  deps?: LoadSessionHistoryDeps,
): Promise<ChatMessage[] | null> {
  // ... path traversal check ...

  // Layer 1: resolve via index
  const resolvedPath = deps?.resolveFilePath?.(sessionId)
  if (resolvedPath) {
    // Layer 2: check content cache
    if (deps?.contentCache) {
      return deps.contentCache.get(resolvedPath)
    }
    // No cache, direct read
    try {
      const content = await fsp.readFile(resolvedPath, 'utf-8')
      return extractChatMessagesFromJsonl(content)
    } catch {
      // Fall through to scan
    }
  }

  // ... existing scan logic (also uses contentCache if available) ...
}
```

The `LoadSessionHistoryDeps` type expands to:

```ts
export interface LoadSessionHistoryDeps {
  resolveFilePath?: (sessionId: string) => string | undefined
  contentCache?: SessionContentCache
}
```

### Layer 3 -- Timeline Response Batching

**Goal:** Allow the client to fetch a timeline page with turn bodies inlined, avoiding
N+1 HTTP requests (one per turn expansion).

**Changes:**

| File | Change |
|------|--------|
| `shared/read-models.ts` | Add `includeBodies: z.coerce.boolean().optional()` to `AgentTimelinePageQuerySchema`. |
| `server/agent-timeline/types.ts` | Add optional `bodies?: Record<string, AgentTimelineTurn>` to `AgentTimelinePage`. Do NOT add `body` to `AgentTimelineItem` (keep items lightweight; bodies live at the page level). |
| `server/agent-timeline/service.ts` | When `query.includeBodies` is true, build the `bodies` map from the page's items and attach to the returned `AgentTimelinePage`. |
| `server/agent-timeline/router.ts` | Parse `includeBodies` from query string and include in the safeParse input object (see integration detail below). |

**Router parsing detail for `includeBodies`:**

The router at `server/agent-timeline/router.ts` manually constructs the query object
for `AgentTimelinePageQuerySchema.safeParse()` (lines 29-33). Add `includeBodies`:

```ts
const query = AgentTimelinePageQuerySchema.safeParse({
  cursor: typeof req.query.cursor === 'string' ? req.query.cursor : undefined,
  priority: typeof req.query.priority === 'string' ? req.query.priority : undefined,
  limit: typeof req.query.limit === 'string' ? Number(req.query.limit) : undefined,
  includeBodies: typeof req.query.includeBodies === 'string' ? req.query.includeBodies : undefined,
})
```

Use `z.coerce.boolean()` in the schema so that the string `'true'` coerces correctly.

**Response shape when `includeBodies=true`:**

```json
{
  "sessionId": "abc-123",
  "items": [
    { "turnId": "turn-2", "sessionId": "abc-123", "role": "user", "summary": "..." }
  ],
  "bodies": {
    "turn-2": {
      "sessionId": "abc-123",
      "turnId": "turn-2",
      "message": { "role": "user", "content": [...] }
    }
  },
  "nextCursor": null,
  "revision": 1234567890
}
```

This is backward-compatible: `bodies` is omitted when not requested.

### ai-router Cache Integration (Optional, Layer 2 Extension)

The `createAiRouter` in `server/index.ts` (lines 426-438) also reads `.jsonl` files
via `codingCliIndexer.getFilePathForSession()` + `fsp.readFile()`. This is a fourth
consumer that benefits from the content cache. However, `readSessionContent` returns
the raw file string (not parsed `ChatMessage[]`), so it cannot directly use
`SessionContentCache` (which stores parsed messages). Two options:

1. **Separate raw-content cache layer** (additional complexity, not recommended for v1).
2. **Leave as-is for now** -- the ai-router reads files less frequently (only on
   explicit user action) and the indexer's O(1) path resolution already eliminates
   the scan overhead. The content cache benefit is smaller here.

**Recommendation:** Leave the ai-router as-is for this PR. Note it as a future
optimization opportunity if profiling shows it as a bottleneck.

## Risk Assessment

**Low risk:**
- Layer 1 is purely additive (new optional parameter, existing behavior preserved).
- Layer 3 is backward-compatible (new optional field).

**Medium risk:**
- Layer 2 cache introduces shared mutable state. Mitigated by:
  - Thorough unit + race condition tests
  - Self-invalidating via stat checks (no stale data even if invalidation hook fails)
  - Request coalescing prevents thundering herd

**Breaking change potential: None.**
All changes are additive. The `loadSessionHistory` function signature adds optional
parameters only. The timeline endpoint adds optional response fields.
