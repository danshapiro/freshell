## SPEC A — CFG-07: persist a stable server-installation identity (and fix the HARNESS-02 conflict)

### A.1 Checklist text (verbatim)

> **CFG-07 — Persist a stable server installation identity.** Reuse one instance ID for the same
> home across restarts and create a different ID for a different home.
> **Playwright validation (`PW-RUST`):** Record the server ID and synced client record, restart
> twice with the same isolated home and assert equality, then launch with a second home and assert
> inequality without losing the first server's tabs.
> — checklist lines 103–106.

The conflict (reconciliation surprise #2, lines 394–398): HARNESS-02's "bite" spec keys on
*"Node persists / Rust regenerates instanceId across a same-home restart."* CFG-07 asks the Rust
server to **persist** (stable across restart) — the exact opposite of what the bite spec asserts
today. Landing CFG-07 makes the bite spec's `rust` branch fail. This spec resolves both.

### A.2 What legacy actually does (the target semantics)

**Two distinct identifiers, not one.** Legacy conflates them under `serverInstanceId` on the wire
but the Rust tree has already split them (`serverInstanceId` + `bootId`). The correct target is to
keep them split and give each its legacy role:

| Identifier | Lifetime | Purpose |
|---|---|---|
| `serverInstanceId` (`srv-<uuid>`) | **Persisted, stable per home** across restarts | Stable *installation* identity — tab-registry keying, live-terminal ownership, session-locator priority, client "am I still talking to the same installation" check |
| `bootId` (`boot-<uuid>`) | **Per-boot**, regenerated every process start | *Restart* signal — a changed `bootId` on (re)connect means the server restarted, so previously-live PTYs are gone |

**Legacy mint/persist path — `server/instance-id.ts`:**

- `loadOrCreateServerInstanceId(baseDir?)` (`server/instance-id.ts:13`): reads
  `<home>/instance-id` (`resolveInstanceIdPath`, line 8–11). If the file exists and is non-empty,
  returns it verbatim (lines 15–17). Only on `ENOENT` does it mint `srv-${randomUUID()}`
  (line 24) and write it atomically (temp file `instance-id.tmp-<pid>-<ts>` + `rename`,
  lines 27–29). Any non-ENOENT read error **throws** (line 20) — a corrupt/unreadable id is fatal,
  not silently regenerated.
- Called once at boot: `server/index.ts:146` → threaded into health payload
  `instanceId` (`server/index.ts:151`), the codex reaper (`:219`), codex runtime factory
  (`:325`,`:359`), and the WS handler (`:384`,`:618`).

**Legacy fallback (only when nothing supplied) — `server/terminal-registry.ts:1300`:**
`this.serverInstanceId = options.serverInstanceId?.trim() || process.env.FRESHELL_SERVER_INSTANCE_ID || \`srv-${process.pid}\``.
`setServerInstanceId()` (`:1310`) lets the WS handler overwrite it once resolved
(`server/ws-handler.ts:625–629`).

**What legacy uses `serverInstanceId` for (all require STABILITY across restart):**

1. **WS `ready` frame** — client records it (`server/ws-handler.ts:625`,`:1851`,`:1878`).
2. **Tab-registry keying** — `server/tabs-registry/types.ts:60` (`serverInstanceId: z.string().min(1)`),
   stamped on every stored record (`server/terminal-registry.ts:2212`,`:2226`,`:2243`,`:2443`).
3. **Sidebar session-locator priority** — `server/sidebar-session-selection.ts:13–26`: a locator
   whose `serverInstanceId` matches the current server scores 3 (highest), `null` scores 2, a
   *different* id scores <2 and is dropped (`:26`). This is how "open sessions on THIS installation"
   win over stale cross-installation locators.
4. **Live-terminal ownership** — `server/ws-handler.ts:2018`,`:2230`:
   `m.liveTerminal?.serverInstanceId === this.serverInstanceId` gates reattachment to a running PTY.

**What the client uses it for (`src/`, frozen reference):**

- `src/lib/ws-client.ts:148–149` reads `ready.serverInstanceId`; exposed via getter `:277`, cleared
  on disconnect `:500`.
- `src/lib/terminal-surface-checkpoint.ts:112`:
  `saved.serverInstanceId !== current.serverInstanceId` → discard a saved terminal surface. This is
  an **"am I on the same installation"** guard (NOT restart detection): if stable across restart,
  saved surfaces survive a restart and are reconciled against the fresh terminal **inventory**
  instead.
- `src/lib/terminal-cursor.ts:282`: `checkpoint.serverInstanceId !== identity.serverInstanceId`
  → discard a replay cursor from a different installation.
- `src/lib/tab-registry-snapshot.ts` keys pane snapshots by `serverInstanceId`
  (`:15`,`:26`,`:76`,`:111`,`:131`).

**How restart is actually detected (both legacy and Rust):** the **terminal inventory** frame.
After a restart the server has zero live PTYs, so the inventory is empty and the client tears down
its ghost panes. The inventory frame already carries `bootId`
(`crates/freshell-protocol/src/server_messages.rs:805`, `TerminalInventory { boot_id, terminals, terminal_meta }`),
so a changed `bootId` is the authoritative "server restarted" edge. **`serverInstanceId` was never
the restart signal in legacy** — it is installation identity.

### A.3 What the Rust server does today

- `crates/freshell-server/src/main.rs:113` — `let server_instance_id = Arc::new(format!("srv-{}", Uuid::new_v4()));`
  **regenerated every boot; no persistence.** This is the CFG-07 gap and the current HARNESS-02
  discriminator.
- `crates/freshell-server/src/main.rs:114` — `let boot_id = Arc::new(format!("boot-{}", Uuid::new_v4()));`
  **already per-boot and correct.**
- Threaded into `WsState` (`main.rs:256` `server_instance_id`, `:257` `boot_id`) and health
  `ApiState.instance_id` (`main.rs:280`, shared `Arc::clone` of `server_instance_id`).
- `WsState` docs already describe both as "stable for the life of this server process"
  (`crates/freshell-ws/src/lib.rs:53–56`).
- Wire shapes already carry both:
  - `Ready { timestamp, boot_id: Option<String>, server_instance_id: Option<String> }`
    (`crates/freshell-protocol/src/server_messages.rs:607–614`).
  - `TerminalInventory { boot_id, terminals, terminal_meta }` (`server_messages.rs:803–807`).
  - Health `"instanceId": &*state.instance_id` (`crates/freshell-api/src/lib.rs:83`).

**Net:** the only behavioral change CFG-07 needs is: make `server_instance_id` **persisted per
home** at `main.rs:113`. `bootId` is already right. No wire-shape change.

### A.4 Target design

Port `server/instance-id.ts` to a small Rust module and consume it at boot.

**New file: `crates/freshell-server/src/instance_id.rs`**

```rust
//! CFG-07: persist a stable server-installation identity keyed on the resolved
//! home. Port of `server/instance-id.ts` (`loadOrCreateServerInstanceId`).
//! Read-once at boot; `bootId` (per-boot) stays the restart signal.

use std::path::{Path, PathBuf};
use uuid::Uuid;

const INSTANCE_ID_FILENAME: &str = "instance-id";

/// `<home>/instance-id` (mirrors `resolveInstanceIdPath`). `home` is the SAME
/// resolved config dir the rest of the server uses (`resolve_home()` →
/// `<home>/.freshell` per the tree's home convention; match `getFreshellConfigDir`).
fn instance_id_path(config_dir: &Path) -> PathBuf {
    config_dir.join(INSTANCE_ID_FILENAME)
}

/// Read the persisted id, or mint + atomically persist a new one. Returns the
/// stable `srv-<uuid>`. A corrupt/unreadable file (non-NotFound error) is
/// FATAL (Err) — never silently regenerated (legacy `instance-id.ts:20` throws).
pub fn load_or_create(config_dir: &Path) -> std::io::Result<String> {
    let path = instance_id_path(config_dir);
    match std::fs::read_to_string(&path) {
        Ok(existing) => {
            let trimmed = existing.trim();
            if !trimmed.is_empty() {
                return Ok(trimmed.to_string());
            }
            // Empty file → fall through and mint (legacy: empty ⇒ create).
        }
        Err(err) if err.kind() == std::io::ErrorKind::NotFound => { /* mint below */ }
        Err(err) => return Err(err), // non-NotFound is fatal, like legacy.
    }

    let id = format!("srv-{}", Uuid::new_v4());
    std::fs::create_dir_all(config_dir)?;
    // Atomic temp-then-rename (legacy `instance-id.ts:27-29`; same discipline as
    // the tree's other atomic writers). Unique temp name avoids concurrent-boot
    // clobber; rename is atomic on the same filesystem.
    let tmp = config_dir.join(format!(
        "{INSTANCE_ID_FILENAME}.tmp-{}-{}",
        std::process::id(),
        std::time::SystemTime::now()
            .duration_since(std::time::UNIX_EPOCH)
            .map(|d| d.as_millis())
            .unwrap_or(0),
    ));
    std::fs::write(&tmp, format!("{id}\n"))?;
    std::fs::rename(&tmp, &path)?;
    Ok(id)
}
```

**Consume at boot — `crates/freshell-server/src/main.rs:113`:**

```rust
// Before:
let server_instance_id = Arc::new(format!("srv-{}", Uuid::new_v4()));
// After (CFG-07): persisted per home; regenerate only if there is no home
// (headless/ephemeral) — matching legacy's baseDir-optional shape.
let server_instance_id = Arc::new(
    home.as_deref()
        .map(|h| /* h is <home> config dir; align with resolve_home() semantics */ instance_id::load_or_create(h))
        .transpose()
        .unwrap_or_else(|err| {
            // Non-NotFound read/write failure: log and fall back to ephemeral so a
            // read-only home can never block boot (parity with logging's boot-tolerance,
            // main.rs:104). This is a documented degradation, not silent regeneration on
            // the happy path.
            tracing::warn!(error = %err, "CFG-07: instance-id persistence failed; using ephemeral id");
            None
        })
        .flatten()
        .unwrap_or_else(|| format!("srv-{}", Uuid::new_v4())),
);
```

### A.5 Fixing HARNESS-02 without weakening what it proves

**What the bite spec proves today** (`test/e2e-browser/specs/harness-02-matrix-bite.spec.ts`):
that the `rust-chromium` project actually boots the **real Rust binary** and is not silently reusing
the Node `TestServer`. It does this by asserting a genuine implementation difference:
`/api/health` `instanceId` **changes** across a same-home restart for `rust`, **stays equal** for
`legacy` (`spec lines 96–104`, keyed on `e2eServerKind`).

**Why CFG-07 breaks it:** once Rust persists, `instanceId` is stable across restart on *both*
kinds → the discriminator collapses (both branches would need `toBe`, so the test no longer
distinguishes the two implementations — a mis-wired rust→Node fixture would now pass).

**The resolution (does NOT weaken the proof):** the bite spec conflates two independent facts.
Split them:

1. **Restart genuinely happened (holds on BOTH kinds after CFG-07):** across a same-home restart,
   `serverInstanceId` is **equal** and `bootId` **differs**. This is the CFG-07 behavior itself and
   belongs in CFG-07's own `PW-RUST` spec — see A.6. It is a correctness assertion, not a
   node-vs-rust discriminator.

2. **The matrix boots the right binary (the bite's real job):** re-key the discriminator to a
   **permanent, structural** node-vs-rust difference that can never accidentally converge — the
   runtime identity exposed by `/api/server-info` (SPEC D). Legacy returns
   `nodeVersion: process.version` (`server/server-info-router.ts:17`); the Rust port (SPEC D, D.4)
   emits `runtime: "rust"` and **no** `nodeVersion`. Assert:

   ```ts
   const info = await (await fetch(`${baseUrl}/api/server-info`)).json()
   if (e2eServerKind === 'rust') {
     expect(info.runtime).toBe('rust')
     expect(info.nodeVersion).toBeUndefined()
   } else {
     expect(typeof info.nodeVersion).toBe('string')   // e.g. "v20.x"
     expect(info.nodeVersion).toMatch(/^v\d+\./)
   }
   ```

   This is **stronger** than the instanceId discriminator: `instanceId` regeneration was an
   incidental gap that CFG-07 legitimately closes; `runtime === "rust"` vs a Node version string is
   an intrinsic property of which binary is running and can never be unified away.

**Decision table — HARNESS-02 bite after CFG-07:**

| Assertion | legacy-chromium | rust-chromium | Proves |
|---|---|---|---|
| `serverInstanceId` equal across same-home restart | ✅ equal | ✅ equal (was ❌) | CFG-07 stability (both) |
| `bootId` differs across restart | ✅ differs | ✅ differs | genuine restart happened |
| `/api/server-info.runtime` | absent / `node`* | `"rust"` | correct binary booted |
| `/api/server-info.nodeVersion` | `"v20.x"` | `undefined` | correct binary booted |

*Legacy server-info does not currently emit `runtime`; the discriminator is the **presence of
`nodeVersion`** (legacy) vs its **absence + `runtime:"rust"`** (Rust). If SPEC D is not yet landed
on the branch when HARNESS-02 is rewired, use the interim fallback below.

**Sequencing / dependency:** the clean discriminator depends on SPEC D's `/api/server-info`.
Ordering options:

- **Preferred:** land SPEC D first, then rewire the bite to server-info runtime identity.
- **Interim fallback (no SPEC D dependency):** key the discriminator on a rust-only `ready`-frame
  field. `ready.bootId`/`ready.serverInstanceId` are `Option` in Rust
  (`server_messages.rs:610–613`) and the Node `ready` frame does not emit `bootId`. Assert the WS
  `ready` frame carries `bootId` only on `rust`. This keeps the bite green through the CFG-07
  landing without waiting on SPEC D, then can be upgraded to the server-info discriminator.

### A.8 Migration notes

- A migrated legacy `.freshell` home already contains an `instance-id` file. Placing the Rust file
  at the **same path** means the Rust server **inherits the legacy installation identity** on
  cutover — existing tab-registry records, session locators, and live-terminal handles keyed on that
  id remain valid. This is the primary reason to match legacy's path exactly (A.4 home note).
- No wire-schema change: `ready`/inventory/health already carry both ids. Clients need no change.
- The one client-visible behavior shift is intended: after a Rust restart on the same home,
  `serverInstanceId` no longer changes, so the client stops discarding saved terminal
  surfaces/cursors on restart and instead reconciles them against the fresh (empty) inventory —
  matching legacy. Confirm no `src/` code treats a *stable* `serverInstanceId`-across-restart as a
  bug (it does not: the surface/cursor guards are installation-scoped, A.2).

### A.9 Risks

- **Home-path mismatch** (highest risk): writing `instance-id` to `<home>` vs `<home>/.freshell`
  wrong breaks legacy-continuity AND diverges from `config.json`/`logs/` location. Verify against
  `resolve_home()` + the legacy `getFreshellConfigDir()` before coding.
- **Concurrent boots on one home** (two servers, same home): temp-then-rename is atomic; last
  rename wins; both end with the same on-disk id on the second read. Acceptable (legacy has the same
  race). Do not add locking here — CFG-02's flock store owns write-serialization for `config.json`,
  not this one-shot id file.
- **Read-only home**: fall back to ephemeral + `warn` (A.4) so boot never blocks — documented
  degradation, mirrors logging's boot-tolerance (`main.rs:104`).
- **HARNESS-02 ordering**: rewiring the bite before SPEC D lands would break it; use the interim
  `ready.bootId` fallback (A.5) or land SPEC D first.

### A.10 NOT to build (fences)

- **Do NOT** persist or rotate `bootId` — it MUST stay per-boot (it is the restart signal).
- **Do NOT** add a config option to toggle persistence — legacy always persists; a toggle is dead
  complexity (and would reintroduce the very ambiguity CFG-07 removes).
- **Do NOT** move restart-detection onto `serverInstanceId` in `src/` — inventory/`bootId` already
  own it; `src/` is frozen reference.
- **Do NOT** fold this into the CFG-02 flock/config store — it is a separate one-shot file with
  different semantics.
- **Do NOT** change the `ready`/inventory/health wire shapes — they already carry both ids.

## SPEC B — SAFE-08: client restore diagnostics + repair

### B.1 Checklist text (verbatim)

> **SAFE-08 — Implement client restore diagnostics and repair.** Record structured diagnostics,
> reconcile stale handles once, and prevent restore loops.
> **Playwright validation (`PW-RUST`):** Seed dead/mismatched terminal and agent handles so the SPA
> reports diagnostics, assert one server repair/recreation and structured log, then reload and
> verify no repeated loop.
> — checklist lines 558–560.

### B.2 What legacy does (the contract to port)

Restore diagnostics ride the `client.diagnostic` WS message and the session-lifecycle log.

**Wire message — `shared/ws-protocol.ts:250` `ClientDiagnosticSchema`:**

```ts
export const ClientDiagnosticSchema = z.object({
  type: z.literal('client.diagnostic'),
  event: z.literal('restore_unavailable'),
  reason: z.literal('dead_live_handle'),
  terminalId: z.string().min(1),
  tabId: z.string().min(1),
  paneId: z.string().min(1),
  mode: z.string().min(1),
  hasSessionRef: z.literal(false),
})
```

**Server handler — `server/ws-handler.ts:1901–1915`:** on `client.diagnostic` with
`event === 'restore_unavailable'`, the server records a structured session-lifecycle event
`kind: 'client_restore_unavailable'` with `terminalId`, `connectionId`, optional `tabId`/`paneId`,
`mode`, `reason`, `hasSessionRef` (`recordSessionLifecycleEvent`, `:1903–1912`). It then `return`s —
**no side effect on the server**; it is purely a diagnostic record.

**Client emit + repair — `src/` (frozen reference):**

- `src/components/TerminalView.tsx:4088–4112`: when a restore is impossible (a `liveTerminal` handle
  the server can no longer honor), the client (a) logs `restore_unavailable` locally (`:4096`),
  (b) sends the `client.diagnostic` frame (`:4100`), and (c) issues a **fresh** `terminal.create`
  carrying `recoveryIntent: 'fresh_after_restore_unavailable'` and registers that request id
  (`addTerminalFreshRecoveryRequestId`, `:4112`).
- `recoveryIntent: 'fresh_after_restore_unavailable'` is a first-class field on `terminal.create`
  (`shared/ws-protocol.ts:271`) and a client type (`src/lib/terminal-restore.ts:10`, used
  `src/App.tsx:842`,`:1008`). The recreate is the **repair**: the dead handle is replaced by one
  fresh terminal, exactly once, driven by the client's own idempotent request id
  (`createRequestId` dedup, the TERM-04 machinery).

**So legacy's SAFE-08 contract has three parts:**

1. **Diagnostics** — client detects a dead/mismatched handle and reports it; server logs a
   structured `client_restore_unavailable` event.
2. **Repair once** — client re-creates the terminal fresh with `recoveryIntent`, deduped by
   `createRequestId` so exactly one recreation happens.
3. **No loop** — because the recreate carries a fresh sessionless intent (`hasSessionRef:false`,
   `recoveryIntent`), the new terminal has no dead handle to re-trigger the diagnostic; the reload
   path resolves against the live registry, not the stale handle.

### B.3 What the Rust server does today

- The wire variant exists: `crates/freshell-protocol/src/client_messages.rs:23` (`#[serde(rename = "client.diagnostic")]`),
  listed in the known-types table (`:79`), with the `restore_unavailable` / `dead_live_handle` /
  `fresh_after_restore_unavailable` consts documented (`:151–191`).
- **But it is never dispatched.** `handle_client_text` in `crates/freshell-ws/src/terminal.rs`
  matches `ClientMessage::…` variants at `:324–438` (TerminalCreate/Attach/Input/Resize/Detach/Kill,
  FreshAgent*, UiScreenshotResult) and the raw `tabs.sync.*` envelopes at `:308–318`. There is **no
  arm for `ClientMessage::Diagnostic`**. A `client.diagnostic` frame either fails the typed parse
  (`:320` `let Ok(message) = … else { return true }`) or, if it parses, hits no handler → **silently
  dropped, no structured log.** That is the SAFE-08 gap: diagnostics are not recorded.
- `recoveryIntent`/`recreate` repair: the `terminal.create` path (`handle_create`, `:324`) already
  accepts the fresh create; verify it tolerates `recoveryIntent` + absent `sessionRef` and does not
  re-attach to the dead handle (the repair already works structurally — the missing piece is the
  **diagnostic record**, and asserting the "exactly once / no loop" invariant).
- Structured logging exists: `crates/freshell-server/src/logging.rs` (DIAG-01) provides
  `tracing`-based JSONL with redaction; WS lifecycle already emits `tracing::info!` events
  (`terminal.rs:121` `ws.connection.established`, `:269–273` `ws.connection.closed`). SAFE-08's
  structured log should ride this same `tracing` sink.

### B.4 Target design

Add a dispatch arm for `client.diagnostic` that emits a structured `tracing` event mirroring
legacy's `client_restore_unavailable` record. The repair (fresh recreate) already exists on the
client and the create path; SAFE-08's server work is the **diagnostic record + the once/no-loop

## SPEC C — SAFE-10: make critical broadcast lag recoverable (resync)

### C.1 Checklist text (verbatim)

> **SAFE-10 — Make critical broadcast lag recoverable.** Resync settings, lifecycle, activity,
> association, materialization, extensions, and tabs instead of leaving a silently stale client.
> **Playwright validation (`PW-RUST`):** Freeze one context with tiny channels while bursting each
> critical event family, resume, and assert a resync/snapshot makes its UI equal the authoritative
> API/second client with no ghost terminals or stale busy state.
> — checklist lines 564–566. (Reconciliation class: **N** — "Phase-2 future.")

### C.2 What the Rust server does today (the gap)

The server→client fan-out is a single `tokio::sync::broadcast` channel:

- Created once: `crates/freshell-server/src/main.rs:150` —
  `broadcast::channel::<String>(1024).0` (capacity 1024 pre-serialized frames).
- Each `/ws` connection subscribes: `crates/freshell-ws/src/lib.rs:348` — `state.broadcast_tx.subscribe()`.
- The per-connection forwarder loop: `crates/freshell-ws/src/terminal.rs:245–262`:

```rust
frame = bcast_rx.recv(), if bus_open => {
    match frame {
        Ok(json) => { /* forward verbatim */ }
        // Slow consumer dropped some frames: the broadcast set is tiny and
        // paced, so this is not expected; skip the gap and keep serving.
        Err(RecvError::Lagged(_)) => {}          // <-- SILENT DROP (terminal.rs:256)
        Err(RecvError::Closed) => { bus_open = false; }
    }
}
```

**The defect:** on `RecvError::Lagged(n)`, tokio has already **dropped `n` frames** for this
receiver, and the loop **silently skips them and keeps serving**. The connection stays open but has
missed events → the client's UI is **permanently stale** (ghost terminals, stuck "busy"/blue
indicators, missed `settings.updated`, missed `sessions.changed`, missed
`freshAgent.session.materialized`, etc.). There is no resync, no reconnect, no signal. This exactly
matches the checklist's "silently stale client."

### C.3 What legacy does (the recovery contract)

Legacy does **not** silently drop. It applies **backpressure and closes** the socket, forcing a
reconnect (which re-runs the full handshake → `ready` + inventory + bootstrap resync):

- `server/ws-handler.ts:1562–1568` `send()` calls `sendJsonMessage(ws, msg, { maxBufferedAmount: this.config.maxWsBufferedAmount, backpressureCloseCode: CLOSE_CODES.BACKPRESSURE, backpressureCloseReason: 'Backpressure' })`.
- `waitForBufferedAmountBelow` (`server/ws-handler.ts:1680–1710`) monitors `ws.bufferedAmount`;
  when a socket exceeds the max buffered bytes it is **closed with the BACKPRESSURE code**, not left
  stale.
- On reconnect the client re-subscribes and re-fetches the authoritative state (the `ready`
  handshake + terminal inventory + `/api/bootstrap`), so the post-reconnect UI is resynced.

So legacy's recovery contract is: **detect the slow/lagged consumer → close → client reconnects →
full resync.** The Rust broadcast model has no per-socket `bufferedAmount` equivalent (tokio drops
oldest frames on lag rather than growing an unbounded buffer), so the Rust port must translate the
*intent* (recover, don't go stale), not the mechanism.

### C.4 Target design

On `RecvError::Lagged`, **trigger a per-connection resync** instead of silently skipping. Two viable
strategies; recommend the in-place resync (surgical, no reconnect flap), with close-and-reconnect as
the documented legacy-equivalent fallback.

**Strategy 1 (recommended) — in-place authoritative snapshot re-push.** On lag, re-push the
authoritative snapshot for every critical event family the checklist names, so the connection
catches up without a reconnect:

```rust
Err(RecvError::Lagged(dropped)) => {
    // SAFE-10: this connection missed `dropped` broadcast frames and would
    // otherwise be permanently stale. Re-push authoritative snapshots so the
    // client converges, instead of the legacy backpressure-close (which the
    // tokio broadcast model can't express as bufferedAmount).
    tracing::warn!(connection_id = conn_id, dropped = dropped, "ws.broadcast.lagged.resync");
    if !resync_after_lag(&mut ws_tx, state, conn_id).await {
        close_reason = "send_error";
        break;
    }
}
```

`resync_after_lag` re-sends, over this connection's outbound path, the current authoritative state
for each critical family (all shapes already exist as server→client frames):

| Family (checklist) | Snapshot source already in tree |
|---|---|
| settings | `settings.updated` from `state.settings` (`WsState.settings`, `lib.rs:58`) |
| lifecycle (terminals) | terminal **inventory** frame from `state.registry` (`TerminalInventory { boot_id, terminals, terminal_meta }`, `server_messages.rs:803`) — carries `bootId` so ghosts reconcile |
| activity / association | current terminal-identity/activity from `state.identity` (`TerminalIdentityRegistry`, `main.rs:219`) |
| materialization | current fresh-agent session snapshots (`fresh_codex`/`fresh_claude`/`fresh_opencode` slices) |
| extensions | `GET /api/extensions` registry snapshot (already serialized, `main.rs:288`) — re-broadcast as its client frame, or rely on the client re-fetching |
| tabs | `state.tabs.query(device_id, client_instance_id)` (`crates/freshell-ws/src/tabs.rs:242`) |

> Implementation note: the connection already owns a single ordered outbound path
> (`conn_tx`/`conn_rx`, `terminal.rs:130`) and the registry can deliver a fresh inventory via the
> same `FrameSink` used for attach/replay (`conn_sink`, `:131`). Re-emit the inventory + settings +
> tabs snapshots through that sink so ordering with in-flight frames is preserved. The identity for
> tabs (`device_id`/`client_instance_id`) is known post-handshake; if not retained on the
> connection, close-and-reconnect (Strategy 2) is simpler for the tabs family specifically.

**Strategy 2 (fallback, closest to legacy) — close with a resync code.** On lag, close the socket
with a dedicated code (e.g. `4010 "resync"`) so the client reconnects and re-runs the full handshake
resync — a direct translation of legacy's BACKPRESSURE close:

```rust
Err(RecvError::Lagged(dropped)) => {
    tracing::warn!(connection_id = conn_id, dropped = dropped, "ws.broadcast.lagged.close");
    let _ = ws_tx.send(Message::Close(Some(CloseFrame {
        code: 4010, reason: "Resync required".into(),
    }))).await;
    close_reason = "broadcast_lagged";
    close_code = Some(4010);
    break;
}
```

The client's existing reconnect path (`src/lib/ws-client.ts`) already re-fetches `ready` + inventory
+ bootstrap, giving a full resync. This is less surgical (a visible reconnect flap) but guaranteed
complete and minimal. **Recommendation:** ship Strategy 1 for the terminal/settings/tabs families it
can cheaply re-push; if the connection does not retain the tabs identity, fall back to Strategy 2
which resyncs *everything* via reconnect. Document whichever is chosen; do not ship a half-resync
that leaves one family stale.

**Also raise the guard against gratuitous lag:** capacity is 1024 (`main.rs:150`). Keep it; the
resync is the safety net for the genuinely-slow consumer, not a license to shrink the channel. The
outer test shrinks the channel via a test hook (see C.7) to force lag deterministically.

### C.7 Tests

**Inner (`crates/freshell-ws`):** unit test that subscribes a receiver to a capacity-2 broadcast,
sends >2 frames before receiving (forcing `Lagged`), and asserts the forwarder path takes the resync
branch (Strategy 1: a resync frame is emitted on the outbound channel; Strategy 2: the loop breaks
with `close_code == Some(4010)`).

**Outer (`PW-RUST`, `rust-chromium`) — new `safe10-broadcast-resync.spec.ts` in `MATRIX_SPECS`:**

1. Boot the owned Rust server with a **tiny** broadcast capacity (`FRESHELL_WS_BROADCAST_CAPACITY=2`).
2. Open two contexts (A and B). **Freeze** A (e.g. `page.evaluate` blocking the WS drain, or CDP
   network throttling) while **bursting** each critical event family from B / the REST API:
   settings change, terminal create/exit (lifecycle), agent activity (busy→idle), session
   association, fresh-agent materialization, extensions, tabs push.
3. **Resume** A.
4. Assert A converges: its terminal list, settings, busy indicators, and tabs equal the
   authoritative `/api/*` state and context B — **no ghost terminals, no stuck busy/blue**.
5. Assert the Rust JSONL log recorded `ws.broadcast.lagged.resync` (S1) or `…lagged.close` (S2) for

### C.8 Risks

- **Beyond-legacy honesty:** legacy relies on reconnect after backpressure; Strategy 1 (in-place
  resync) is *beyond legacy*. That is acceptable and better UX, but flag it in the commit/PR as an
  intentional improvement, not a claimed parity. Strategy 2 is the true legacy-equivalent.
- **Resync completeness:** a partial resync (missing one family) still leaves a stale corner and
  would fail step 4. Either re-push **all** named families (S1) or reconnect-resync **everything**
  (S2). No middle ground.
- **Ordering:** re-push snapshots through the connection's existing ordered outbound sink so a
  resync frame can't interleave incorrectly with a normal in-flight frame.
- **Thundering resync:** if many connections lag simultaneously (server-wide burst), N resyncs fire
  at once. Acceptable for the critical-event volume (paced), but do not re-push on every frame — only
  on the `Lagged` edge.

### C.9 NOT to build (fences)

- **Do NOT** shrink the production broadcast capacity to "test lag" — force lag via the test-only
  env hook.
- **Do NOT** silently continue on `Lagged` (the current bug) — a stale client is the failure SAFE-10
  exists to prevent.
- **Do NOT** attempt a per-frame `bufferedAmount` port — the tokio broadcast model drops rather than
  buffers; translate the *intent* (resync), not the mechanism.
- **Do NOT** resync on `Closed` — that path is shutdown; leave it as `bus_open = false`.
- **Do NOT** build a new per-family delta protocol — reuse the existing authoritative snapshot
  frames (inventory, settings.updated, tabs query).

### D.8 Risks

- **Redaction completeness is the acceptance crux** (step 4): a redactor that only knows fixed key
  names misses the "unknown future secret-marked field." The marker-based rule (D.4) is what makes
  the test pass; do not ship key-name-only redaction.
- **Post-serialize regex is fragile**: redact the structured `Value` before serialization, not the
  output string (avoids partial matches / nested-object corruption).
- **`arch` spelling** differs from Node (`x86_64` vs `x64`) — expected; the outer test must assert
  against the host truth, not Node's spelling.
- **Ownership boundary**: `logging.rs` is a concurrently-frozen slice — copy/adapt its scrub
  approach into the diag module rather than importing internals.
- **Live `wsConnections` counter**: if the tree currently mints connection ids without a live count,
  add an atomic counter; a stale count fails step 3.

- **Do NOT** add live-control side effects to `/api/debug` or `/api/server-info` — they are
  read-only; only `/api/perf` mutates.
- **Do NOT** invent extra debug fields beyond legacy's shape (version, appVersion, wsConnections,
